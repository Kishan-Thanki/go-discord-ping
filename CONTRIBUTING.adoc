= Contributing to go-discord-ping

First off, thank you for considering contributing to `go-discord-ping`! It's people like you that make open-source software such a great community.

== Development Setup

. *Clone the repository:*
+
[source,bash]
----
git clone https://github.com/Kishan-Thanki/discord-ping.git
cd discord-ping
----

. *Set up your environment:* +
  Create a `.env` file in the root directory:
+
[source,env]
----
TOKEN=your_discord_bot_token
----

. *Run the bot:* +
  We provide a standard `Makefile` for easy development.
+
[source,bash]
----
make run
----

== Architectural Guidelines

This bot is specifically engineered for *High Performance* and *Zero-Allocation*. Please adhere to the following strict guidelines when submitting Pull Requests:

=== 1. String Concatenation

*NEVER* use `fmt.Sprintf` or the `+` operator inside of loops when building strings.
*ALWAYS* use `strings.Builder`.

[source,go]
----
// ❌ BAD (Causes memory allocations)
desc := ""
for _, user := range users {
    desc += user.Name + "\n"
}

// ✅ GOOD (Zero-allocation)
var desc strings.Builder
for _, user := range users {
    desc.WriteString(user.Name)
    desc.WriteString("\n")
}
----

=== 2. Database Interactions

* Do not modify the `PRAGMA` settings in `database/db.go`. The bot relies heavily on SQLite `WAL` mode.
* Always use *Prepared Statements* (`sql.Stmt`) to prevent SQL Injection. Do not manually format SQL strings.

=== 3. Goroutines & State

* If a command relies on asynchronous wait times (like `!remind`), use `time.AfterFunc` rather than blocking `time.Sleep`.
* Use `sync.Map` or `sync.Mutex` when interacting with global state to prevent race conditions during high traffic.

== Pull Request Process

. Ensure your code compiles: `make build`
. Ensure your code passes all static analysis: `make lint`
. Submit your PR! Our GitHub Actions CI pipeline will automatically verify your code.
